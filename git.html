Git Tutorial: Mastering Version Control for Beginners

Unleash the power of Git version control with our beginner-friendly tutorial.

Setup

Download Git for your operating system:

  • Git for OSX
  • Git for Windows
  • Git for Linux

Create a New Repository

Create a new directory, open it, and perform a git init to create a new Git repository.

Checkout a Repository

Create a working copy of a local repository by running the command git clone /path/to/repository. For a remote server, use git clone username@host:/path/to/repository.

Workflow

Your local repository consists of three "trees" maintained by Git:

  • Working Directory - Holds the actual files.
  • Index - Acts as a staging area.
  • HEAD - Points to the last commit.

Add & Commit

Propose changes (add to the Index) using git add <filename> or git add *. Commit these changes using git commit -m "Commit message". The file is now committed to the HEAD, but not yet in your remote repository.

Pushing Changes

Your changes are now in the HEAD of your local working copy. To send those changes to your remote repository, execute git push origin master. Change master to the branch you want to push your changes to.

Branching

Branches are used to develop features isolated from each other. The master branch is the "default" branch when you create a repository. Use other branches for development and merge them back to the master branch upon completion.

Create a new branch named "feature_x" and switch to it using git checkout -b feature_x. Switch back to master using git checkout master and delete the branch with git branch -d feature_x. A branch is not available to others unless you push it to your remote repository using git push origin <branch>.

Update & Merge

Update your local repository to the newest commit by executing git pull in your working directory to fetch and merge remote changes.

To merge another branch into your active branch (e.g. master), use git merge <branch>. If there are conflicts, resolve them manually by editing the files shown by Git. After resolving, mark them as merged with git add <filename>.

Tagging

It's recommended to create tags for software releases. Create a new tag named 1.0.0 by executing git tag 1.0.0 1b2e1d63ff. The 1b2e1d63ff stands for the first 10 characters of the commit ID you want to reference. Get the commit ID by looking at the log.

Log

Study repository history using git log.

Add parameters for customized logs:

  • Commits by a certain author: git log --author=bob
  • Compressed log, one line per commit: git log --pretty=oneline
  • ASCII art tree of all branches: git log --graph --oneline --decorate --all
  • Files changed: git log --name-status

For more options, see git log --help.

Replace Local Changes

If you made an error, replace local changes using git checkout -- <filename>. This replaces changes in your working tree with the last content in HEAD. Added changes or new files are kept.

To drop all local changes and commits, fetch the latest history from the server and point your local master branch to it: git fetch origin, git reset --hard origin/master.

Useful Hints

Built-in Git GUI: gitk.

Colorful Git output: git config color.ui true.

Show log with one line per commit: git config format.pretty oneline.

Use interactive adding: git add -i.
